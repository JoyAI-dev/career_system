# 活动系统交互式测试辅助

参数: `$ARGUMENTS`（可选，如用户数量或特定活动类型）

**重要**: 严格按照以下阶段执行，每个阶段必须等用户确认后才进入下一阶段。

---

## Phase 0: 开场说明（必须首先执行）

在做任何事情之前，先向用户清楚介绍本次测试的角色分工和交互方式：

```
你好！我是活动测试辅助。我们将一起测试活动系统（圆桌会议、探索搭子等）。

先说明一下我们的分工：

┌─────────────────────────────────────────────────┐
│  👤 你的角色                                     │
│  ─────────────                                  │
│  • 你会操控 1 个测试用户                          │
│  • 在浏览器中登录并手动操作                       │
│  • 给我发指令，告诉我让其他用户做什么              │
│  • 你是总指挥，我只在你指示后才行动                │
├─────────────────────────────────────────────────┤
│  🤖 我的角色                                     │
│  ─────────────                                  │
│  • 我会操控另外 5 个测试用户                      │
│  • 通过后台浏览器自动操作（你看不到）              │
│  • 每完成一步，我会向你汇报状态                    │
│  • 我不会主动行动，完全听你指挥                    │
├─────────────────────────────────────────────────┤
│  🔄 交互方式                                     │
│  ─────────────                                  │
│  1. 我先准备好 6 个测试账号                       │
│  2. 你选一个角色，我操控剩余 5 个                  │
│  3. 你在浏览器操作你的角色                        │
│  4. 你发指令（如 "让所有人加入圆桌会议"）          │
│  5. 我执行后汇报结果，等你的下一步指令             │
│  6. 如此循环，直到测试完成                        │
└─────────────────────────────────────────────────┘
```

展示完毕后，进入 Phase 1 收集信息。**可以将 Phase 0 的说明和 Phase 1 的问题合并在同一条消息中发出**，减少来回次数。

## Phase 3: 启动浏览器 Subagent

### 3.1 架构规则（严格遵守）

- **主进程（本进程）**: 只做与用户的沟通和协调，**绝不**直接操作浏览器
- **subagent**: 每个 AI 用户一个独立的 `agent-browser` subagent
- 使用 `Task` tool 的 `subagent_type: "agent-browser"` 启动
- 使用 `run_in_background: true` 保持主进程畅通
- 每个 subagent 记录其 `agent_id` 以便后续 resume 继续操控

### 3.2 启动 5 个 Subagent（后台）

为每个 AI 用户启动一个后台 subagent，prompt 模板：

```
你是活动测试的自动化操作员，操控用户 "{用户名}"。

1. 打开 https://career-staging.joysort.cn/login
2. 输入用户名: {用户名}，密码: {密码}
3. 点击登录
4. 确认进入 Dashboard 页面
5. 截图当前页面作为就绪证明
6. 报告："{显示名} 登录成功，已进入 Dashboard，等待下一步指令。"

完成以上步骤后停止，等待 resume 继续。
```

并行启动所有 5 个 subagent，等待全部登录成功。

### 3.3 汇报就绪状态

所有 subagent 就绪后，向用户汇报：


## Phase 4: 指令执行循环

### 4.1 进入等待模式

主进程进入等待模式，持续接收用户指令。**不要主动假设下一步操作**，等用户明确告诉你。

### 4.2 常见指令处理

| 用户说 | 执行方式 |
|--------|----------|
| "让所有人加入圆桌会议" | resume 5 个 subagent，依次点击加入 |
| "让 XXX 先加入" | 仅 resume 对应 subagent 执行加入 |
| "等我先加入" | 暂停，等用户说"好了"再继续 |
| "让其他人也加入" | resume 剩余未加入的 subagent |
| "让 XXX 安排时间" | resume 组长 subagent 执行排期 |
| "让 XXX 完成活动" | resume 对应 subagent 标记完成 |
| "让 XXX 和 YYY 做 ZZZ" | 仅 resume 指定的 subagent |
| "截图给我看看 XXX 的页面" | resume 对应 subagent 截图 |
| "刷新所有人的页面" | resume 所有 subagent 执行刷新 |

### 4.3 组长加入逻辑

基于 Phase 1 的 `USER_ROLE` 决定：

**如果 USER_ROLE = "leader"（用户当组长）**:
1. 提示用户先在浏览器中加入圆桌会议
2. 等用户确认 "我加入了"
3. 再 resume AI_USERS 的 subagent 依次加入

**如果 USER_ROLE = "member"（用户当组员）**:
1. resume AI_USERS[0] 的 subagent 先加入（AI_USERS[0] 成为组长）
2. 提示用户可以加入了
3. 等用户确认后，resume 剩余 AI 用户加入

### 4.4 操控 Subagent 的 prompt 模板

执行操作时，resume subagent 并传入指令：

```
继续操控用户 "{显示名}"。

当前任务：{具体操作描述}
例如：点击"圆桌会议"卡片上的"加入"按钮，确认加入成功。

操作步骤：
1. {步骤1}
2. {步骤2}
3. 截图当前页面
4. 报告操作结果

完成后停止，等待下一步指令。
```

### 4.5 状态反馈模板

每次操作完成后，主进程向用户汇报：

```
✅ 已完成：
  - {显示名}: {操作结果}（第N个加入，{角色}）

⏳ 等待指令：
  - {显示名}: 已就绪
  - {显示名}: 已就绪

👤 你 ({HUMAN_USER.显示名}): {当前状态/下一步建议}

请告诉我下一步操作。
```

### 4.6 错误处理

如果某个 subagent 操作失败：
1. 截图失败页面
2. 向用户报告：`❌ {显示名} 操作失败：{错误描述}`
3. 询问用户：重试 / 跳过 / 手动检查

## Phase 5: 活动流程参考（供操控时参考）

### 圆桌会议（首个活动，GROUP_6）

```
Dashboard → 看到"圆桌会议"卡片
  → 点击"加入" → 系统创建/加入虚拟组
  → 第一个加入的人 = 组长 (LEADER)
  → 6 人全部加入 → 状态变为 FULL
  → 组长安排时间 → 状态变为 SCHEDULED
  → 开始活动 → IN_PROGRESS
  → 完成活动 → COMPLETED → 解锁"探索搭子"
```

### 活动类型链（前置解锁关系）

```
1. 圆桌会议 (GROUP_6, 6人, 组长标记完成)
   └→ 2. 探索搭子 (PAIR_2, 2人, 自选配对)
      └→ 3. 深度探索 (PAIR_2, 2人)
         └→ 4. 竞赛 (CROSS_GROUP, 12人, 两组对抗)
            └→ 5. 试水案例 (INDIVIDUAL, 1人)
               └→ 6. 聚焦讨论 (GROUP_6, 6人)
                  └→ 7. 导师拍卖 (GROUP_6, 仅竞赛获胜组)
```

### 关键 UI 元素提示

- **Dashboard 活动卡片**: 显示活动类型名称、当前成员数/所需人数、加入按钮
- **加入按钮**: 可能是 "加入" 或 "参加" 文字
- **活动详情页**: 成员列表、组长标记、时间安排、完成按钮
- **完成模式**: 圆桌会议是 `LEADER_ONLY`，只有组长能标记完成

---

## 注意事项

1. **Seed 幂等**: 多次运行清除旧数据重新创建，安全可重复
2. **Staging 环境**: 所有操作在 `https://career-staging.joysort.cn`，不影响生产
3. **顺序控制**: 加入活动时严格控制顺序（决定组长），不要并行加入
4. **灵活响应**: 用户可能随时改变计划，不要假设后续步骤
5. **操作失败截图**: subagent 失败时必须截图反馈
6. **主进程畅通**: 主进程只做沟通协调，所有浏览器操作由 subagent 执行
7. **subagent 复用**: 使用 `resume` 复用已有 subagent，不要每次新建
8. **并行启动**: 5 个 subagent 的初始登录可以并行启动（无竞态），后续操作按指令决定是否并行
